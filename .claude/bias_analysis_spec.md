# 馬場バイアス分析 改善仕様(mamahuhu.app)

## 現状
- 各競馬場ごとに「前開催日の12レース」を集計して馬場バイアスを表示している。
- この「直近の均質なデータを最重視する」方針は維持する(鮮度・条件の均質性・分かりやすさが長所)。

## 改善の全体方針
直近データ(前開催日)を主役にしたまま、長期ベースラインで少サンプルの暴れを補正する。
細かくクロス集計しすぎず、階層フォールバックでサンプル不足に対応する。

## 追加で必要なデータ項目
1. **コース設定(柵の位置: A/B/Cコース)** — 開催ごと。JRA公式の開催情報から取得
2. **馬場状態** — レースごと(良/稍重/重/不良)
3. **トラック種別**(芝/ダート)と**距離** — 既存になければ追加
4. (任意・将来)コーナー通過順位 — 脚質バイアス算出用

## 集計ルール

### 直近バイアス(メイン表示)
- 対象: 前開催日の12レース(現行どおり)
- ただし**コース替わり(例: A→B)があった場合は、それ以前のデータと混ぜない**
- 芝とダートは必ず分けて集計する

### 長期ベースライン(補正用)
- 対象期間: **直近2〜3年**
- **路盤改修・大規模コース変更があった競馬場は、その時点でデータをリセット**(例: 京都は2023年改修後のみ)
- 用途: 直近12レースの推定値を、長期傾向に向けて縮小(ベイズ的補正)。サンプルが少ないほど長期側に寄せる

### セグメント(区切り)の粒度
細かく区切りすぎるとセルあたりのレース数が激減するため、以下の粗さで束ねる:
- **距離**: 実距離ではなくカテゴリで束ねる(例: 短距離 / マイル〜中距離 / 長距離)。理想は「最初のコーナーまでの距離」が近いコース同士をまとめる
- **馬場状態**: まず2区分(良 / 道悪=稍重以下)。4区分はサンプル不足
- **階層フォールバック**: 細かいセル(場×トラック×距離カテゴリ×馬場状態)にデータが足りない場合、一段粗いセル(場×トラック全体など)の値で代用する

## 内外(枠)バイアスの定義

### グループ分け
枠番ではなく**相対位置 = 馬番 ÷ 出走頭数** で3分割する:
- 内: 相対位置 ≦ 1/3
- 中: 1/3 < 相対位置 ≦ 2/3
- 外: 相対位置 > 2/3

頭数に応じて自動で伸縮する(16頭立て: 1〜5番が内 / 10頭立て: 1〜3番が内)。
表示ラベルには枠番を併記してもよいが、集計は相対位置で行う。

### 指標
- 複勝率ではなく**正規化着順(着順 ÷ 出走頭数)の平均**を全体期待値と比較する
- 少頭数レースを混ぜても歪まず、12レース程度の小サンプルに強い

## 表示上の要件
- サンプル数(対象レース数・頭数)を明示する
- 確度の目安を併記する(例: 「12レース中8レースで内有利」のような不確実性込みの表現)
- コース替わり直後は「バイアスがリセットされた可能性が高い」旨を注記する

## 将来的な拡張(優先度低)
- 直近側の加重平均(前開催日×2、前々開催日×1 など)
- 脚質バイアス(前残り/差し届く)をコーナー通過順位から算出し、枠バイアスと併せて表示
- 長期ベースライン側も古い年を軽く加重し、馬場管理方針の緩やかな変化に追従

# 予想セクション 仕様(期待値ベース)

## 基本思想
「よく当たる予想」ではなく**「オッズより高い確率を見つける予想」**を作る。
市場(オッズ)は非常に優秀な予測器なので、ゼロからモデルを作らず**市場確率をベースに、バイアス情報で補正する**構造にする(Bill Benter型アプローチ)。

## アルゴリズム構成

### 1. 市場確率の算出
- 単勝オッズから市場の暗黙確率を逆算する: `p_market ≈ 0.8 ÷ 単勝オッズ`(控除率約20%を除去)
- より正確にはレース内で正規化する: 各馬の (1÷オッズ) を合計で割る

### 2. バイアス補正モデル
- **条件付きロジットモデル**(レース単位で1着馬を選ぶ多項選択モデル)を推奨。実装が難しければ馬単位のロジスティック回帰でも可
- 基礎項: log(p_market) — 係数をほぼ1に固定またはそのまま学習
- 補正特徴量(すべて予想時点で計算可能なもののみ):
  - 相対枠位置(内/中/外) × 当日推定バイアス(内外)
  - 脚質 × 当日推定バイアス(前残り/差し) ※脚質データ導入後
  - バイアスのサンプル数・確度(信頼度が低い日は補正を弱める)
  - 馬場状態、コース設定、距離カテゴリ
- 学習データ: 過去レース結果 + その時点で計算できたバイアス値

### 3. 買い目の選定
- 期待値 = モデル確率 × オッズ
- **期待値 > 1.1〜1.2 の馬のみ推奨として表示**する(閾値は要バックテスト)
- 該当馬がいないレースは「見送り」と表示する。見送りが大半になるのが正常な挙動

### 4. 資金管理(表示する場合)
- **分数ケリー(ケリー基準の1/4程度)**で推奨賭け金比率を算出
- フルケリーは推定誤差に脆弱なため使わない

## 評価・バックテスト要件

### 評価指標
- **的中率は主指標にしない**。以下を使う:
  - **ROI**(回収率): 推奨買い目に従った場合の収支
  - **キャリブレーション**: モデル確率X%の馬が実際にX%勝っているかを確率帯ごとに検証(信頼度曲線)
  - 対数損失(log loss)での市場確率との比較: 市場に勝てているかの直接検証

### リーク防止(最重要)
- 学習・検証は**時系列分割**(過去で学習→未来で検証)。ランダム分割は禁止
- 特徴量は**予想公開時点で入手可能だったデータのみ**を使用:
  - オッズは実際に賭けられるタイミングのもの(前日オッズ or 発走N分前)を使う。確定オッズでのバックテストは成績を過大評価する
  - バイアス値も「その日の朝時点で計算できた値」(=前開催日まで)で再現する
- 検証期間は最低1年以上。季節・開催場の偏りを避ける

## 表示上の要件
- 推奨馬には「モデル確率」「オッズ由来の市場確率」「期待値」を併記する
- 「見送り」判断も明示する(買わないことが価値である旨)
- 免責: 予想は情報提供であり的中・収益を保証しない旨を表示

## 実装の優先順位
1. 市場確率の算出とレース内正規化
2. バックテスト基盤(時系列分割・リーク防止)を先に作る
3. 最小の補正モデル(枠位置×バイアスのみ)で検証
4. 特徴量を段階的に追加(脚質、確度重み付けなど)し、都度ROI・キャリブレーションで判断

# 運用構成(GitHub Actions / DB契約なし)

## 基本方針
外部DBを契約せず、**GitHub内で完結**させる。データはファイルとしてリポジトリにコミットして永続化する。

## データ保存
- **SQLiteファイル**(推奨)または年別のJSON/CSVをリポジトリ内に保存する
- 規模感: JRAは年間約3,500レース。2〜3年分でも数十MB程度で、GitHubの1ファイル100MB制限内に収まる
- 生データ(レース結果・オッズ)と集計済みデータ(バイアス値・予想)は分けて保存する
- 集計済みJSONはサイトから直接参照できる場所に出力する

## ワークフロー構成

### スケジュール実行
```yaml
on:
  schedule:
    - cron: '0 22 * * 0'  # 月曜 7:00 JST(cronはUTC指定なのでJST-9時間)
  workflow_dispatch:        # 手動実行を必ず併記(scheduleの遅延・スキップ対策)

permissions:
  contents: write           # GITHUB_TOKENでcommit&pushするために必要

concurrency:
  group: data-update        # 二重実行によるコミット競合を防止
  cancel-in-progress: false
```

### 処理フロー
1. リポジトリをcheckout
2. 開催日のレース結果・オッズ・馬場状態・コース設定をスクレイピング
3. SQLite(または生データファイル)に追記
4. バイアス集計・予想モデルの推論を実行し、集計済みJSONを生成
5. `git commit & push`(変更がない場合はスキップして正常終了)

### 実装上の要件
- **冪等性**: 同じ日に何度実行してもデータが重複・破損しない設計にする(レースIDでUPSERT)
- **失敗時の再実行**: workflow_dispatchで任意の日付を指定して取り直せるようにする(input引数で対象日を受け取る)
- スケジュールは開催の有無に関わらず起動し、開催がなければ何もせず終了する

## サイトへのデータ供給
Netlifyに自動デプロイされる様、設定済み

## 既知のリスクと対策

### 1. cronのタイムゾーン
- GitHub ActionsのcronはUTC。JSTから9時間引いて指定する

### 2. scheduleの遅延・スキップ
- 混雑時に数十分遅延、稀に実行されないことがある
- 対策: workflow_dispatchの併記 + 冪等な処理 + 翌日にも保険で再実行するスケジュールを追加してもよい

### 3. 海外IPブロック(検証済み)
- **検証結果: netkeibaはGitHub Actionsから取得可能。JRA公式はブロックされる**
- したがって**全データをnetkeibaから取得する**(下記「データソース」参照)。JRA公式へのアクセスは実装しない
- 将来netkeiba側がブロックを始めた場合の代替案: 自宅PC・ラズパイ等をセルフホストランナーとして登録する(GitHub標準機能・無料)

### 4. スクレイピングのマナー
- リクエスト間隔を空ける(1秒以上)、User-Agent明記、robots.txt・利用規約の確認
- 週1回の取得なので負荷は軽微だが、リトライは指数バックオフで実装する

## データソース(netkeiba一本化)

全データをnetkeibaから取得する。JRA公式は使わない(Actionsランナーからブロックされるため)。

### 取得対象とnetkeiba上の所在
- **レース結果(着順・馬番・頭数)**: レース結果ページ(race/レースID)
- **単勝オッズ**: 結果ページの確定オッズ。まず確定オッズで開始する(バックテスト上の注意は予想セクション参照)
- **馬場状態(良/稍重/重/不良)・天候**: レース結果ページのレース情報欄
- **トラック種別・距離・回り**: 同上(例:「芝右1600m」をパース)
- **コーナー通過順位**: レース結果ページ下部の通過順位欄(脚質バイアス用)
- **コース設定(A/B/C柵)**: ⚠️ **要確認**。netkeibaのレースページで柵設定が取得できるか実装時に検証する。取得できない場合の代替:
  - 開催回次(例: 3回中山)と日次から柵替わりパターンを別途マスタ化して紐づける
  - 柵設定が取れない間は「同一開催内かどうか」でリセット判定を代用する(開催が替われば柵も変わっている可能性が高い)

### レースIDの規則
netkeibaのレースIDは `年(4桁)+競馬場コード(2桁)+回次(2桁)+日次(2桁)+レース番号(2桁)` の12桁。日付と開催からIDを機械的に生成でき、一括取得・冪等なUPSERTのキーとしてそのまま使える。

### 注意事項
- netkeibaのHTML構造変更に備え、パース失敗時は**エラーで停止して通知**する(壊れたデータを保存しない)
- 一部データ(タイム指数等)はプレミアム限定。本仕様で必要なデータは無料ページの範囲で揃う想定
- 文字コードはEUC-JPのページが残っているため、取得時にエンコーディングを明示的に処理する

## 実装の優先順位
1. 過去2〜3年分の一括取得スクリプト(レースID生成 → 結果・オッズ・通過順位のパース)
2. コース設定(柵)情報がnetkeibaから取れるかの確認、取れなければ代替ルールを実装
3. 週次更新ワークフロー(schedule + workflow_dispatch)
4. 集計済みJSONのサイト連携
